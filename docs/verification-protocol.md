# Verification Protocol

**버전:** v2.0
**작성:** 2026-04-14 (v1 → v2)
**목적:** Claude(CC + Claude.ai 채팅방 공통)가 "완료"를 주장할 자격을 구조적으로 통제해, 거짓 완료와 의도 어긋남을 줄인다.
**적용:** 이 문서를 읽는 Claude는 이하 규칙을 세션 내내 지킨다.

---

## 0. 우선순위 및 개정 규칙

### 0.1 이 문서의 우선순위
이 문서의 규칙은 이 세션의 다른 어떤 규율보다 **우선**한다.
다만 사용자의 **명시적 세션 내 지시**(예: "이번만 프로토콜 생략")는 존중한다.

### 0.2 이 문서 자체의 개정 규칙 (anti-loop)
이 프로토콜은 **실제 사용 데이터 없이 재개정되지 않는다.**
- 재개정 조건: 최소 **1주일 실사용** 후
- 재개정 트리거: 구체적 실패 사례 3건 이상 기록
- 금지: 사용 없이 "더 잘 만들 수 있을 것 같다"는 직관만으로 개정
- **이유:** 메타 작업이 본 작업을 갉아먹지 않도록. 프로토콜 설계에도 프로토콜이 필요함.

## 1. 세션 시작 — 모드 선언

### 1.1 모드 종류

모든 세션은 다음 3개 모드 중 **하나를 사용자가 선언**한 뒤에만 진행한다.
선언 없이 작업 요청 시, Claude는 **요청 동사로 자체 판정 후 1줄 명시**한다 (§1.3). 사용자가 틀렸다 싶으면 수정.

| 모드 | 상징 | 의미 |
|---|---|---|
| **🔴 프로덕션** | 출시 가능 품질 | 빌드·린트·테스트 통과 필수. §3 증거 매트릭스 엄격. |
| **🟡 프로토타입** | 탐색적 구현 | "추상 느낌 → 테스트 버전" 워크플로. 증거 매트릭스 완화. 버릴 수 있음을 전제. |
| **🟢 탐색** | 대화·질문·설계 | 코드 작성 없음. 검증 블록 불필요. |

### 1.2 모드 전환 규칙

- **🟡 → 🔴 승격:** 기존 체크리스트 항목을 🔴 기준으로 재검증 필수. 재검증 통과 전에는 새 항목 금지.
- **🔴 → 🟡 강등:** 사용자가 "이 부분은 프로토타입"이라 명시한 범위만 강등. 이유를 체크리스트에 주석.
- **🟢 → 🟡/🔴:** 직전 논의 내용을 체크리스트 형태로 **재진술 후 확정**. "방금 얘기한 대로" 시작 금지.

### 1.3 자동 판정 규칙 (모드 미선언 시)

- 질문형 / "설명해줘" / "~할 수 있어?" → 🟢 탐색
- "만들어줘" / "수정해줘" / "추가해줘" → 🟡 프로토타입 (기본값)
- "배포" / "프로덕션" / "출시" 언급 → 🔴 프로덕션
- 판정을 응답 첫 줄에 표시: `[판정: 🟡 프로토타입] — 수정해줘 요청이라 이렇게 판단. 다르면 알려줘.`

## 2. 태스크 시작 — 규모 판정 및 체크리스트

### 2.0 태스크 규모 판정 (⭐ v2 신규)

Claude는 응답 첫 줄에 **태스크 규모**도 함께 선언한다.

| 규모 | 기준 | §2.1~2.3 요구 |
|---|---|---|
| **Micro** | 1파일 내 1~20줄, 새 로직 없음 (오타/색/상수/삭제) | 체크리스트·가정 섹션·스케치 모두 생략. §6.1 축약 검증 블록만 필수. |
| **Small** | 1~3파일, 로직 변경 있으나 아키텍처 결정 없음 | 체크리스트·가정 섹션 필수. 스케치 선택. §6.2 표준 블록. |
| **Standard** | 그 이상 | 체크리스트·가정 섹션·스케치(🟡 시) 모두 필수. §6.2 표준 블록. |

사용자가 판정을 되돌릴 수 있음. 모호하면 더 큰 규모로 판정하는 것이 기본 (under-enforcement < over-enforcement).

### 2.1 체크리스트 제안 (Small / Standard)

```
## 체크리스트 초안 (확정 필요)
- [ ] 항목 1 — [구체 산출물 or 행동]
- [ ] 항목 2 — ...

## 명시되지 않아 내가 가정한 것
- 가정 A — [왜 이렇게 가정했는지]
- 가정 B — ...

**이대로 진행할까? 빠진 항목이나 수정할 부분 있어?**
```

**가정 섹션이 진짜로 비어있다면** "가정 없음 — 요청이 완전히 구체적"이라고 명시. 자명한 것까지 나열하지 말 것 (가정 인플레이션 금지).

### 2.2 확정 전 코드 금지

사용자의 명시적 확정("맞아", "진행해", 수정 후 "이대로") 전에는 파일 수정 금지.
**탐색적 읽기**(Read/Glob/Grep)는 제안 전에도 허용.

## 3. 증거 × 태스크 매트릭스

완료 주장 시 다음 증거를 태스크 유형에 따라 제시.

| 태스크 유형 | 🔴 프로덕션 필수 증거 | 🟡 프로토타입 완화 증거 |
|---|---|---|
| **로직/기능 수정** | (1) 변경 diff (2) 빌드 로그 (exit 0) (3) 관련 테스트 결과 | (1) diff (2) dev 구동 확인 or 빌드 통과 |
| **UI 컴포넌트** | (1) diff (2) 빌드 통과 (3) Figma 토큰 대조표 | (1) diff (2) 스크린샷 or 렌더 확인 |
| **DB 스키마** | (1) 마이그레이션 파일 (2) 타입 재생성 (3) RLS 정책 | (1) 마이그레이션 파일 |
| **리팩토링** | (1) diff (2) 빌드 통과 (3) 전체 테스트 통과 (regression 0) | 🟡 금지 — 리팩토링은 🔴 전용 |
| **설정·문서** | diff만 | diff만 |

### 3.1 증거 형식 규약 (⭐ v2 신규 — 핵심 강화)

증거는 **기계·사용자가 1분 내 대조 가능한 형식**만 인정:

**✅ 인정:**
- 파일 경로 + **줄번호 범위** + **실제 내용 3~5줄 인용**
  ```
  증거: src/components/LoginForm.tsx:42-45
  ```typescript
    <Link href="/forgot-password" className="text-sm text-gray-500">
      비밀번호 찾기
    </Link>
  ```
  ```
- 실제 명령 stdout **마지막 N줄** (exit code 포함) + **해당 Bash 도구 호출이 세션에 실존해야 함**
  ```
  증거: $ npm run build
  > ✓ Compiled successfully
  > exit 0
  ```

**❌ 불인정:**
- "빌드 통과했습니다" (서술)
- "src/foo.ts" (경로만, 줄번호·인용 없음)
- "테스트 다 돌려봤어요"

### 3.2 항목 ↔ 증거 1:1 강제

체크리스트 각 항목은 **자기를 직접 입증**하는 증거를 가진다.
여러 항목이 같은 파일을 참조할 땐 줄 번호로 구분.

### 3.3 증거 부족 시
증거를 제시할 수 없으면 "완료"가 아니라 §5 "검증 불가 지점"으로 보고.

## 4. 프로토타입 모드 — Sketch-First (Standard 규모만)

🟡 + Standard 규모에서 코드 작성 전 스케치 필수:

```
## 스케치
**접근:** [1~2줄]
**핵심 파일:**
- `path/to/file.ts` — [역할]

**데이터 흐름:** [의사코드 or 3~5줄]
**의도적으로 생략:** [에러 처리, 엣지 케이스 등]

**이 방향 맞아?**
```

**Small 이하에서는 스케치 생략 가능.** 대신 코드 + 선택지 2~3개를 동시 제시("A: 모달 / B: 새 페이지 / C: drawer — 어느 쪽?") 형태 허용.

## 5. 합법적 정지 — "검증 불가 지점"

### 5.1 사용 조건 (남용 방지)
§5 선언 전에 **최소 2회의 해결 시도** 필요. 각 시도는 실제 도구 호출 증거로 남아야 함.

### 5.2 보고 포맷
```
## 검증 불가 지점 도달
**완료된 것:** [체크리스트 항목 + 증거]
**막힌 지점:** [어떤 항목에서 왜]
**시도 내역:** [실제 명령·수정 각각 1줄 + 결과]
**사용자가 해주면 재개 가능한 것:** [구체적 요청]
**판단이 필요한 것:** [선택지]
```

### 5.3 우회 금지
에러 시 테스트 수정, any 캐스팅, 기능 묵음 제거 등 금지.

## 6. 검증 블록 (완료 시 필수)

### 6.1 축약 블록 (Micro 규모)
```
<verification>
mode: 🟡 | scale: Micro | task: [1줄]
diff: [파일:줄범위] — [3~5줄 인용]
build: [exit 0 / N/A + 이유]
</verification>
```

### 6.2 표준 블록 (Small / Standard)
```
<verification>
mode: [🔴 | 🟡] | scale: [Small | Standard]
task: [1줄]

checklist:
  - [x] 항목 1 — 증거: [경로:줄범위, 필요시 인용]
  - [x] 항목 2 — 증거: ...
  - [ ] 항목 3 — 검증 불가: [§5 참조]

evidence:
  build: [exit 0 / exit N / 미수행+이유]
  tests: [pass N/M / 미수행+이유]
  diff_files:
    - path/to/file.ts:12-28
    - path/to/other.tsx:44-67

unchecked_assumptions:
  - [확인 못 한 가정]

self_doubt:
  - [가장 확신 못 하는 항목 1개 이상 — "모두 확신" 금지]

next_steps_if_any:
  - [남은 작업 있으면]
</verification>
```

### 6.3 블록 없이 "완료" 금지
블록 없거나 미해결 체크리스트 있는데 "완료" 말하면 → 사용자가 즉시 거부해야 함.

### 6.4 채팅방 모드 한계 명시 (⭐ v2 신규)
**채팅방에서 이 프로토콜은 *정직한 실수*를 잡지 *의도적 기만*을 잡지 못한다.**
CC 훅이 실제 도구 호출 기록과 대조할 때만 기만 탐지 가능.

## 8. 의도 어긋남 피드백

### 8.1 현재 세션 처리
사용자가 "이건 내가 원한 게 아냐" 류 불만:
1. 즉시 수정 X. 먼저 **어떤 부분이 어긋났는지 되묻기** or 가설 확인.
2. 파악된 어긋남을 체크리스트 주석으로 기록:
   ```
   - [x] 항목 N — 증거: ... // ⚠ 어긋남: [요약] → 수정 후 재검증
   ```
3. 다음 태스크 가정 섹션에 반영.

### 8.2 크로스 세션 학습 — INTENT_LOG (⭐ v2 신규)

**같은 부류 교정이 2회 이상 반복**되면:
1. Claude가 **"이 패턴을 INTENT_LOG에 추가할까?"** 제안
2. 확정 시 프로젝트 루트의 `CLAUDE.md` 또는 `.claude/INTENT_LOG.md`에 등록:
   ```markdown
   ## Intent Corrections Log
   - [UI] 링크는 새 페이지 아니라 모달 선호 (2026-04-14)
   - [Code] try/catch는 swallow 아니라 console.error + rethrow (2026-04-14)
   ```
3. 세션 시작 시 Claude는 이 파일을 Read하여 가정 섹션에 반영.
4. 등재 후 **3세션 자연 적용** 시 CLAUDE.md 본문 규칙으로 승격 → 로그에서 제거.

**경계:** 1회 교정은 기록 X (과잉 학습 방지).

## 9. 금지 행동

- ❌ 모드 선언 없이 코드 작성 (자동 판정은 §1.3에서만 허용)
- ❌ Small/Standard에서 체크리스트 확정 없이 코드 작성
- ❌ Standard 🟡에서 스케치 없이 코드 작성
- ❌ 증거 없이 "완료" 주장
- ❌ 자기 보고 문자열을 증거로 사용 (§3.1)
- ❌ 경로만 쓰고 줄번호·인용 누락 (§3.1)
- ❌ "모두 확신" self_doubt (§6.2)
- ❌ 테스트 통과시키려 테스트 수정
- ❌ 타입 에러를 any로 우회
- ❌ 에러 묵음 (try/catch empty)
- ❌ 지시 범위 밖 파일 수정
- ❌ §5 남용 (시도 2회 미만 선언 금지)

## 9.5 컨텍스트 리프레시 프로토콜 (⭐ v2.1 추가)

대화가 길어지면 컨텍스트 창이 무거워져 성능 저하·혼선 발생. 리셋이 필요하나 리셋하면 메모리 손실. 해법: **챕터 단위로 리셋 + md 파일 바통 패스.**

### 9.5.1 Claude의 의무 — 챕터 종료 감지 및 안내

Claude는 다음 신호 하나라도 감지되면 **스스로 사용자에게 리셋 제안**:

- 🔴/🟡 모드에서 `<verification>` 블록이 통과되어 한 태스크가 확실히 종료됨
- 주제가 크게 전환됨 (예: UI 작업 → DB 스키마)
- 세션 누적 tool call이 많아 응답이 체감 느려짐
- 사용자가 **"다음 것 하자"** / **"이제 다른 거"** 류 전환 신호를 냄

안내 문구 예시:
```
✅ 챕터 종료 감지: [직전 태스크 요약]
다음 작업 전에 컨텍스트 리프레시를 권장합니다.

- CC에서: `/clear` 입력
- 채팅방에서: 새 대화 열기 (Project 안에서)

재개 시 읽을 파일:
- `verification-protocol.md`
- `bsmg-v5/CLAUDE.md`
- `bsmg-v5/.claude/RESUME.md` ← 내가 방금 업데이트함
```

### 9.5.2 바통 패스 — RESUME.md 규약

챕터 종료 시 Claude는 프로젝트 루트의 `.claude/RESUME.md`를 **생성 또는 갱신**하고 안내에 포함:

```markdown
# RESUME — [프로젝트명]
**최종 갱신:** [YYYY-MM-DD HH:MM]

## 방금 끝낸 것
- [태스크 요약]
- 관련 파일: [경로:줄범위]

## 다음에 할 것
- [구체 다음 작업]
- 선결 조건: [있으면]

## 열린 질문
- [사용자 결정 대기 중인 것]

## 참고
- [핵심 컨텍스트 파일 경로들]
```

### 9.5.3 리셋 후 재개 절차

새 세션 시작 시:
1. `verification-protocol.md` 로드 (규율 재확인)
2. 프로젝트 `CLAUDE.md` 로드
3. `.claude/RESUME.md` Read → "방금 끝낸 것 / 다음 할 것" 파악
4. `INTENT_LOG`(있으면) Read → 누적 선호 반영
5. 사용자에게 **"RESUME.md를 읽었어. `[다음 할 것]`부터 진행하면 돼?"** 확인

### 9.5.4 사용자의 권한
사용자가 **"아직 리셋 안 해"** 하면 무시하고 계속 진행. Claude의 안내는 제안일 뿐 강제 아님.

### 9.5.5 Claude.ai Project 동기화 알림 (⭐ v2.2 추가)

CC 세션에서 **Tier 1 파일**이 변경됐으면, 챕터 종료 안내에 반드시 포함:

**Tier 1 파일 (Project knowledge에 올라가 있는 것):**
- `verification-protocol.md`
- 각 프로젝트의 `CLAUDE.md`
- 각 프로젝트의 `PROJECT_MAP.md` (있으면)
- Project에 올라간 기타 spec/규율 문서

**알림 포맷 예시:**
```
🔄 Claude.ai Project 동기화 필요
다음 파일이 이 세션에서 변경됨 — 해당 Project knowledge에서 **삭제 후 재업로드** 필요:
- [Project명]: verification-protocol.md, bsmg-v5/CLAUDE.md
- [Project명]: lens-docs/PROJECT_MAP.md

사유: Claude.ai Project knowledge는 자동 동기화 안 됨.
```

**변경 없으면** 이 알림 생략. 불필요한 노이즈 방지.

**Tier 2 파일** (`RESUME.md` 등)은 동기화 불필요 — 어차피 다음 세션에 드래그로 전달.
**Tier 3 파일** (훅 코드, INTENT_LOG 등)도 동기화 불필요 — 채팅방에서 안 씀.
